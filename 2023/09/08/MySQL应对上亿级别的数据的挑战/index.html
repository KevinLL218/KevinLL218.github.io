MySQL应对上亿级别的数据的挑战

稍微更新一点吧，分成两部分，一个是学习MySQL的时候知道的，MySQL Cluster，另一个就是之前用过的MyCat。

NDB

NDB的定义就是一个in-memory storage engine, 和普通的innodb和MyISM不同的是in-memory，和memory不同的是persistence。

NDB作为MySQL原生的框架，其配置过程和拓扑结构相对比较复杂。

MyCat with MQ

关于集群中使用innodb还是ndb引擎，官方手册里面是有过讨论的（MySQL :: MySQL 8.0 Reference Manual :: 25.2.6 MySQL Server Using InnoDB Compared with NDB Cluster），简单来说，按需使用。

实践过的组合是InnoDB组成集群+MQ和MyCat

MyCat是一个Alibaba开源的中间件，虽然出自ali，存在各种问题，但是应对一般情况是可以接受的。而且这个工具也是可以替代的。

MQ则是为了数据同步，elasticsearch，以及限流削峰而引入的，如果没有es，则是可以直接使用NDB搭建集群。使用replica技术作为备份。为了降低系统的复杂度而使用了MQ，即从Write数据库中监控binlog，读取变动后一份发送给es，一份发送给replica服务器作为Reader Client，而MyCat则是在Write数据库与Connector中间添加的一层dispatcher中间件。

SQL发送到Client之前先经过MyCat服务器进行解析，以一个Insert命令为例，MyCat服务器根据预先定义的规则，计算出这个SQL语句应该在哪一台服务器上执行，然后转发给目标数据库。

这样的结果就是，原本1亿条数据可以分摊到多台服务器上，实践上则是十台服务器，每台服务器10个分片，每个分片承担1百万条数据。再加上建立合适的索引，可以极大提高系统的反应速度。

app 与 mycat中间也是有redis缓存层，正是这一层redis，正好解决了MySQL分布式的一个难点: 数据重复

duplication

项目使用了一个统一的数据网关作为外界数据输入入口，然后通过服务调用，处理输入的数据，但是这样会存在一个问题: 两条数据路由到两台服务器上处理，处理时同时向同一个数据库里插入，或同时修改同一个数据。这样的情况比较多发，因此又引入了redisson分布式锁，当处理一条数据时，锁住处理进程，其他服务器无法继续，只能挂起等待。最终避免冲突。

这样的解决方法并不好，锁的粒度太大了，后来把锁提前到了数据网关处，利用时间窗口来限制每一个step的数据量，对于数据库的访问，则使用了retry策略，如果插入失败，则等待几秒后再次执行，虽然不能完全避免，但是比之前速度提高了很多。

ES引入

另外一个优化则是数据查询的优化，原来的查询导出都是通过MySQL数据库直接查询，这样的结果就是当导出的数据达到十万以上时，会出现连接超时的现象，虽然可以修改配置文件提高超时时间，但是这反而会影响其他操作的效率: 长时间持有一个连接，使得pool中的可用连接变少，甚至会影响正常的业务逻辑。

当然后来使用了读写分离策略，最后还是由于导出的需求比较多，经过讨论下还是引入了ES作为查询导出的数据源。

ES天然适合查询，尤其是在数亿条数据中查询大量结果集时，ES比SQL更快。所以针对大量查询导出的需求引入了ES。

由于ES重建index时会固定花费1秒的时间，这个延迟无法避免，再加上MQ生产的速度不可预测，消费的速率固定，因此严重时会存在几个小时的延迟，这也是当时情况下的弊端。

导出的功能又重新设计，由原来的实时下载转为定时任务。问题是系统是内部系统，对于安全性要求较高，不能使用OBS等服务，生成的文件暂时保存在服务器上，随着时间增长会不断压缩磁盘空间。这也是后来出了导出模块其他都上了docker的原因。生成的文件因为要做审计，文件要保留一段时间。

定时任务则是通过lts-job来实现。定时执行方法，方法会去查询保存的任务信息，定时去执行查询导出任务。由于不需要做前端展示，速度会明显快很多。
